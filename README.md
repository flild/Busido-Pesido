# Busido-Pesido

Проект Busido-Pesido — это веб-сайт для специалиста по поведению, состоянию и благополучию животных (собаки и кошки). Сайт предоставляет информацию об услугах, принципах работы, позволяет пользователям записываться на консультации (бронировать услуги) и читать тематические статьи.

Также в проекте реализована закрытая административная панель для отслеживания заявок и управления контентом блога.

## Технологический стек

- **Фреймворк:** [Next.js 15+](https://nextjs.org/) (App Router)
- **Язык:** TypeScript
- **Стилизация:** Чистый CSS (с использованием современных свойств и переменных)
- **База данных:** [better-sqlite3](https://github.com/WiseLibs/better-sqlite3) (локальная SQLite БД)
- **Графики:** [Recharts](https://recharts.org/) (для дашборда в админке)
- **Иконки:** [lucide-react](https://lucide.dev/)
- **Рендеринг контента:** [react-markdown](https://github.com/remarkjs/react-markdown) (для статей блога)
- **Анимации:** `motion` (framer-motion)

## Структура проекта

### `/app`
Основная директория маршрутизации Next.js:
- `/` — Главная страница с презентацией услуг и подходов.
- `/about`, `/support`, `/complex-cases`, `/cats`, `/professionals`, `/library` — Информационные страницы услуг и форматов.
- `/booking` — Страница записи с интерактивной формой.
- `/free-consultations` — Страница записи на бесплатные консультации (1-7 числа каждого месяца).
- `/blog` — Блог со списком статей.
- `/blog/[slug]` — Страница отдельной статьи блога (Markdown рендеринг).
- `/admin` — Главная страница административной панели (дашборд и статистика).
- `/admin/applications` — Управление заявками.
- `/admin/articles` — Управление статьями блога (создание, редактирование, удаление).
- `/api` — API-роуты для работы с базой данных (заявки, статьи, статистика).

### `/components`
Реиспользуемые React-компоненты:
- `BookingForm.tsx` — Форма заявки на услуги.
- `BlogList.tsx` — Компонент списка статей с фильтрацией.
- Интерактивные компоненты главной страницы: `ApproachTabs`, `StateSlider`, `IssueCard`, `Navigator`, и др.

### `/lib`
Вспомогательные утилиты и инициализация БД:
- `db.ts` — Настройка и подключение к SQLite. Содержит схему базы данных.

### `/data`
Директория, создаваемая автоматически, в которой хранится файл базы данных `sqlite.db`.

## База данных (SQLite)

В проекте используются две основные таблицы:

1. **`articles`** — Статьи блога:
   - `id` (INTEGER, PK)
   - `title` (TEXT)
   - `slug` (TEXT, UNIQUE) — URL-идентификатор
   - `summary` (TEXT) — Краткое описание
   - `content` (TEXT) — Текст статьи в формате Markdown
   - `category` (TEXT) — Системная категория для фильтрации
   - `tag` (TEXT) — Отображаемый тег на карточке
   - `status` (TEXT) — Статус публикации ('published', 'draft')
   - `created_at` (DATETIME)

2. **`applications`** — Заявки клиентов:
   - `id` (INTEGER, PK)
   - `service` (TEXT) — Выбранная услуга
   - `date` (TEXT) — Дата записи
   - `time` (TEXT) — Время записи
   - `name` (TEXT) — Имя клиента
   - `email` (TEXT) — Email
   - `contact` (TEXT) — Контакт для связи (Telegram, телефон)
   - `request_text` (TEXT) — Описание проблемы/запроса
   - `status` (TEXT) — Статус обработки ('new', 'contacted', 'completed', 'cancelled')
   - `created_at` (DATETIME)

## Административная панель (`/admin`)

Админка предназначена для управления бизнесом (ИП) без сложных систем интеграции (т.к. проект небольшой):

- **Дашборд:** Показывает общую статистику: всего заявок, новые заявки, конверсия в работу (переход из 'Новая' в 'Завершена'), количество опубликованных статей. Отрисованы графики динамики заявок по дням и распределение по популярным услугам (через Recharts).
- **Управление заявками:** Таблица со всеми поступившими заявками. Возможность изменения статуса (воронка) и удаления.
- **Управление статьями (CMS):** Полноценный CRUD для блога. Создание новых статей, редактирование существующих, удаление. Поддержка Markdown.

## Стилизация (`/app/globals.css`)

Проект  использует Tailwind. 

## Развертывание

Так как проект использует SQLite, при развертывании (например, в Docker) необходимо убедиться, что папка `/data` примонтирована как постоянный том (Persistent Volume), чтобы база данных не сбрасывалась при перезапуске контейнера. В serverless окружениях (Vercel) локальный SQLite работать не будет в режиме записи, потребуется миграция на облачную БД (например, Cloud SQL или Turso/LibSQL), если планируется деплой туда.

# Контекст проекта: Busido-Pesido (Next.js 15 App Router + SQLite)

**Суть:** Разрабатываем сайт для зоопсихолога (SSR/SSG блог + лендинг + закрытая админка для управления статьями и заявками). Проект изначально сгенерирован ИИ, поэтому содержит много наивного кода, который мы постепенно рефакторим под production-ready состояние.

## ⚠️ Главные правила разработки (ОБЯЗАТЕЛЬНО К ИСПОЛНЕНИЮ)

1. **Никакого "оптимистичного" UI.** Редиректы (`router.push`) и обновление стейтов происходят *только* после получения ответа `res.ok` от сервера.
2. **Жесткая валидация.** Бэкенд никому не доверяет. Обязательная проверка на пустые строки (`trim()`), валидация ENUM-значений (статусы) и проверка `changes === 0` при `UPDATE`/`DELETE`.
3. **Обработка SQL-ошибок.** На бэке обязательно перехватываем уникальные ограничения (например, `SQLITE_CONSTRAINT_UNIQUE` для слагов статей) и отдаем понятный статус `409 Conflict`.
4. **Data Fetching в клиентских компонентах.** Для Next.js 15 функции загрузки данных не выносятся из `useEffect` наружу без крайней необходимости. Для обновления данных используется паттерн со стейт-триггером `refresh`, чтобы избежать проблем с линтером и каскадными рендерами.

## ✅ ЧТО УЖЕ СДЕЛАНО И РАБОТАЕТ СТАБИЛЬНО

1. Инфраструктура и БД
lib/db.ts: Настроено кэширование соединения с better-sqlite3 через globalThis, чтобы Next.js не блокировал базу (database is locked) при hot-reload. Включен режим WAL.

Безопасность (middleware.ts): Настроена базовая авторизация. Закрыты роуты /admin/* и все /api/*, кроме клиентского POST-запроса на заявку. Логины и пароли вынесены в .env (ADMIN_USER, ADMIN_PASSWORD), добавлена блокировка (fail-secure 500) при отсутствии переменных окружения.

2. Бэкенд (API-роуты)
/api/applications & /[id]: Настроена серверная валидация полей заявки. Защита от мусорных статусов. Отдача 404, если заявка не найдена.

/api/articles & /[id]: Валидация контента, проверка уникальности URL (slug) через отлов ошибки 409.

/api/stats: Защищено от падений при пустой БД (установлены fallback-значения).

3. Клиентская часть (Админка и Формы)
BookingForm.tsx: Блокировка кнопки при отправке (isSubmitting), обработка ошибок (Toast) и очистка формы через useRef.

Дашборд и CMS: Отлов ошибок fetch, защита графиков от пустых массивов. Дожидание ответа сервера при смене статусов, удалении заявок и сохранении статей. Вывод ошибок от БД прямо в UI.

4. Интерактивные компоненты главной страницы
use-mobile.ts: Устранен Hydration Mismatch, переписано на безопасный useSyncExternalStore.

StateSlider.tsx, ApproachTabs.tsx: Проведена оптимизация памяти (данные вынесены из стейтов, используется useMemo), добавлена строгая типизация (TabId).

Navigator.tsx: Внедрена жесткая try/catch валидация структуры данных при чтении из localStorage, исключены падения UI при битых данных.

5. Публичный блог и SEO
SSR Блога: Списки статей и страницы блога переведены на серверный рендеринг без клиентских fetch. Внедрена жесткая фильтрация черновиков на уровне SQL (AND status = 'published').

Markdown: Подключен плагин remark-gfm для поддержки таблиц и зачеркиваний.

SEO: sitemap.ts генерируется динамически из базы. robots.ts и sitemap отвязаны от хардкода через NEXT_PUBLIC_BASE_URL. В компоненты (карусель отзывов) интегрирован next/image.

Глобальная обработка ошибок: Настроены файлы app/error.tsx (перехват 500-х ошибок) и app/not-found.tsx (элегантная обработка 404).

### 6. Миграция на Tailwind CSS и Рефакторинг UI (Завершено)
* **Дизайн-система (`tailwind.config.ts`)**: Создана конфигурация Tailwind, включающая всю кастомную палитру (matcha, forest, caramel, rose и др.), сложные градиенты (dopamine-gradient) и анимации (floatBlob, pulseHint). 
* **Глобальные стили (`globals.css`)**: Очищен от хардкода. Оставлен только фундамент (`@layer base`) для типографики и переиспользуемые UI-компоненты (`@layer components`) для кнопок, бейджей и базовых карточек.
* **Перевод компонентов на утилиты**: Полностью переписаны на Tailwind ключевые секции: 
  * `app/page.tsx` (Сложные фоновые градиенты и псевдоэлементы).
  * `Header.tsx` (Мобильная навигация и адаптив).
  * Интерактивные компоненты (`ApproachTabs`, `FactorCloud`, `BlogList`, `CaseInteractive`, `BookingForm`, `FormatsSection`, `FreeConsultationsDemo`, `Footer`, `LibraryInteractive`, `IssueCard`).
* **Очистка от антипаттернов**: Избавились от привязки стилей к структуре DOM (`nth-child`). Теперь все цвета карточек и вкладок явно управляются через конфигурационные массивы внутри React-компонентов.
